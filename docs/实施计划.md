# 实施计划

## 目标

项目先验证底层网络方案，再开发控制平面和产品界面。首发版本聚焦 Linux/Docker 服务端、Windows/Linux 客户端、Windows/Linux 实施端，Android 和 Windows 服务端放到后续阶段。

整个实施过程遵循一个原则：底层网络能力尽量复用成熟开源项目，项目自研重点放在服务端管理、角色权限、网段映射、客户端体验和部署交付。

## 阶段一：开源底座验证

### 目标

确认是否可以基于 EasyTier 作为底层网络能力。验证内容不涉及最终界面，只验证关键网络能力是否满足项目需求。

### 工作内容

```text
1. 搭建 Linux 测试服务端。
2. 部署 EasyTier 或候选底层组件。
3. 准备 Windows 客户端测试机。
4. 准备 Linux 客户端测试机。
5. 准备 Windows 实施端测试机。
6. 验证 TUN 虚拟网卡是否正常工作。
7. 验证 ICMP、TCP、UDP 是否可转发。
8. 验证子网代理能力。
9. 验证 Docker 部署可行性。
```

### 验证场景

```text
服务端：Linux + Docker
客户 A：Windows，真实网段 192.168.1.0/24
客户 B：Linux，真实网段 192.168.1.0/24
实施端：Windows
```

需要重点验证两个客户真实网段相同的情况。若底层项目无法直接解决冲突，需要确认是否可以在控制层或转发层增加映射逻辑。

### 验收标准

```text
1. 实施端可以 ping 通客户侧映射地址。
2. 实施端可以访问客户侧 TCP 服务。
3. 实施端可以访问客户侧 UDP 服务。
4. NTP 或等价 UDP 测试可正常返回。
5. Windows 和 Linux 路由添加、删除正常。
6. 断线重连后网络可恢复。
7. Docker 服务端部署路径明确。
```

### 阶段产出

```text
1. EasyTier 是否适合作为底座的结论。
2. 关键命令和配置记录。
3. 网络验证结果。
4. 替代方案判断：NetBird / Nebula 是否需要进入下一轮验证。
```

## 阶段二：总体架构定型

### 目标

在 PoC 结果基础上确定最终技术路线。此阶段要明确哪些能力复用开源项目，哪些能力由本项目自研。

### 工作内容

```text
1. 确定底层网络组件。
2. 确定服务端控制平面结构。
3. 确定客户端、实施端和服务端的统一程序形态。
4. 确定映射网段池和实施端地址池。
5. 确定节点、网段、权限、地址池的数据结构。
6. 确定 Windows 和 Linux 的系统权限处理方式。
7. 确定 Docker 部署方式。
```

### 关键设计

服务端至少包含以下模块：

```text
1. 节点管理模块
2. 网段管理模块
3. 映射地址池模块
4. 权限管理模块
5. 隧道连接管理模块
6. 路由下发模块
7. 后台 API
8. Web 管理后台
```

客户端和实施端至少包含以下模块：

```text
1. 服务端连接模块
2. TUN / 虚拟网卡模块
3. 本机网段识别模块
4. 路由配置模块
5. 断线重连模块
6. 角色模式模块
7. 桌面界面模块
```

### 阶段产出

```text
1. 技术架构说明。
2. 数据结构说明。
3. 服务端 API 草案。
4. 客户端与服务端交互流程。
5. 开源底座改造范围说明。
```

## 阶段三：服务端基础能力

### 目标

先把中心管理能力搭起来，使节点可以注册、上线、上报网段，并获得服务端分配的映射网段。

### 工作内容

```text
1. 建立服务端项目结构。
2. 实现节点注册。
3. 实现节点在线状态维护。
4. 实现角色识别：服务端 / 普通客户端 / 实施端。
5. 实现网段登记。
6. 实现映射网段分配。
7. 实现实施端地址分配。
8. 实现基础权限模型。
9. 实现后台 API。
10. 实现 Docker Compose 部署。
```

### 数据表

初版至少需要以下数据：

```text
nodes
segments
mapped_segments
operator_addresses
access_policies
address_pools
connection_sessions
```

### 阶段产出

```text
1. 可运行的 Linux 服务端。
2. 可运行的 Docker 服务端。
3. 节点注册 API。
4. 网段上报 API。
5. 映射网段分配逻辑。
6. 权限查询逻辑。
```

## 阶段四：客户端基础能力

### 目标

实现 Windows/Linux 客户端连接服务端、识别本机网段、上报网段、参与转发。

### 工作内容

```text
1. 实现客户端启动模式。
2. 实现连接服务端。
3. 实现本机网卡扫描。
4. 识别物理网卡、虚拟网卡、VPN 网卡。
5. 展示候选网段。
6. 支持用户选择网段。
7. 支持填写网段名称和备注。
8. 上报网段到服务端。
9. 接收服务端分配的映射网段。
10. 启动底层隧道转发。
```

### Windows 重点

```text
1. 管理员权限提示。
2. 虚拟网卡驱动处理。
3. 路由添加和删除。
4. 防火墙提示或规则放行。
5. 程序退出时清理路由。
6. 系统服务或开机自启动。
```

### Linux 重点

```text
1. /dev/net/tun 权限。
2. ip route 配置。
3. iptables 或 nftables 配置。
4. systemd 服务。
5. IP forwarding 配置。
```

### 阶段产出

```text
1. Windows 客户端命令行版本。
2. Linux 客户端命令行版本。
3. 网段扫描和上报功能。
4. 客户端参与转发能力。
5. 客户端日志和基础诊断能力。
```

## 阶段五：实施端基础能力

### 目标

实现 Windows/Linux 实施端连接服务端后自动获得实施端地址，并访问所有已登记映射网段。

### 工作内容

```text
1. 实现实施端启动模式。
2. 连接服务端并申请实施端地址。
3. 接收可访问网段列表。
4. 自动添加映射网段路由。
5. 显示客户、网段名称、真实网段、映射网段。
6. 支持 ping 测试。
7. 支持 TCP 连通性测试。
8. 支持 UDP 连通性测试。
9. 支持断线重连后恢复路由。
```

### 默认权限

实施端初版默认访问所有已登记网段。后台后续可以增加实施端权限范围限制，但不作为首发阻塞项。

### 阶段产出

```text
1. Windows 实施端命令行版本。
2. Linux 实施端命令行版本。
3. 实施端地址分配。
4. 映射网段路由下发。
5. ICMP/TCP/UDP 访问验证。
```

## 开发顺序建议

项目不应先做完整界面，而应先把网络链路跑通。推荐顺序如下：

```text
1. EasyTier PoC
2. 服务端数据模型
3. 节点注册和网段上报
4. 映射网段分配
5. 实施端访问映射网段
6. 普通客户端默认隔离
7. 后台权限配置
8. Windows/Linux 命令行版本
9. 后台管理界面
10. 桌面界面
11. 安装包和部署脚本
12. Android 版本
```

这样可以避免前期在界面上投入过多，而底层网络能力尚未确认。

## 首次开发建议

正式开发前，建议先建立一个最小可运行原型：

```text
1. 一个 Linux 服务端。
2. 一个 Windows 客户端。
3. 一个 Windows 实施端。
4. 客户端上报 192.168.1.0/24。
5. 服务端分配 100.64.1.0/24。
6. 实施端访问 100.64.1.10。
7. 验证 ping、TCP、UDP。
```

这个原型跑通后，再加入第二个客户，并让第二个客户也使用 `192.168.1.0/24`。如果两个重复真实网段都能通过不同映射网段访问，说明项目核心设计成立。

## 暂定技术栈

技术栈需要根据 PoC 结果最终确定。当前建议如下：

| 模块 | 暂定方向 |
|---|---|
| 底层网络 | 优先验证 EasyTier，备选 NetBird / Nebula |
| 服务端 | Go 或 Rust，根据底层项目语言决定 |
| 后台 API | REST 或 gRPC |
| 数据库 | SQLite 起步，后续可扩展 PostgreSQL |
| Web 后台 | Vue / React |
| 桌面端 | Tauri 优先评估 |
| Android | 原生 Android + VPNService |
| 部署 | Docker Compose + Linux 原生服务 |

如果基于 EasyTier 二开，Rust 和 Tauri 的组合更自然；如果选择 NetBird，Go 体系更自然。

## 开源许可证处理

选型阶段必须检查许可证：

```text
1. EasyTier：LGPL-3.0
2. NetBird：需要关注 AGPL-3.0 相关目录
3. Nebula：MIT
4. ZeroTier：MPL-2.0 与 nonfree 目录并存
5. Headscale：BSD-3-Clause
```

如果项目后续要闭源商业化，不能只看功能适配度，还要评估许可证是否允许当前分发方式和二开方式。

## 当前结论

当前最合理的推进方式是先用 EasyTier 做 PoC。若 EasyTier 的子网代理、UDP 转发、Windows/Linux 虚拟网卡和 Docker 部署均满足要求，就以它作为底层网络能力基础；若网段冲突映射或控制能力难以改造，再引入 NetBird 或 Nebula 做第二轮评估。

项目正式开发应从“网络链路验证”开始，而不是从界面开始。界面、后台和权限配置建立在底层链路可控之后，才能减少返工。

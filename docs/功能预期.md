# 功能预期

## 项目定位

本项目拟开发一款面向多局域网互通场景的组网软件。它的核心目标是让分布在不同网络环境中的电脑、服务器、虚拟机网段和现场设备网段，通过统一的中心服务端建立可控连接，使实施人员能够稳定访问被授权的远端设备。

软件采用一个本体、三种角色的设计。程序启动后可选择服务端、客户端或实施端，不拆成三套独立软件。底层网络能力优先基于成熟开源项目进行改造，避免从零开发虚拟网卡、隧道、NAT 穿透和三层转发能力。

## 建设背景

客户现场、实施电脑、服务器和虚拟机环境经常处在不同局域网中。传统做法通常依赖端口映射、远程桌面、临时 VPN、跳板机或人工改路由，配置成本高，稳定性和可维护性也不理想。

本项目要解决的主要问题包括：

```text
1. 不同客户现场可能使用相同真实网段。
2. 实施人员需要访问多个客户网段。
3. 普通客户端默认不应访问其他客户网段。
4. 客户端可能存在多个物理网卡、虚拟机网卡和 VPN 网卡。
5. 需要支持 ping、TCP、UDP 等三层网络访问。
6. 服务端需要集中维护节点、网段、权限和映射关系。
```

前期暂不把安全体系作为重点，不展开加密、审计、强认证、零信任策略等设计。当前立案阶段重点讨论功能形态、网络模型、开源底座和实施路径。

## 角色定义

| 角色 | 部署位置 | 核心职责 |
|---|---|---|
| 服务端 | 云服务器、中心机房、Linux 服务器、Docker 环境 | 管理节点、网段、映射关系、权限和转发入口 |
| 客户端 | 客户现场 Windows/Linux 电脑、网关机、工控机 | 上报本地可访问网段，并按权限参与转发 |
| 实施端 | 实施人员 Windows/Linux/Android 设备 | 连接服务端后访问被授权的远端映射网段 |

服务端、客户端和实施端在产品上属于同一个软件。启动时可以通过界面选择角色，也可以通过命令参数指定模式，例如：

```text
net-tool --mode server
net-tool --mode client
net-tool --mode operator
```

桌面版可以提供三个入口：

```text
服务端
客户端
实施端
```

## 核心网络模型

项目采用三层虚拟组网模型。底层通过 TUN 虚拟网卡接管 IP 包，支持 ICMP、TCP 和 UDP 转发。

已确认的网络能力如下：

```text
1. 支持 ICMP，用于 ping。
2. 支持 TCP，用于 Web、RDP、SSH、数据库、接口服务等。
3. 支持 UDP，用于 NTP 和其他 UDP 协议。
4. 暂时不支持广播和组播。
5. 暂时不承诺跨网段自动发现设备。
6. 远端设备统一通过映射 IP 访问。
```

采用 TUN 模式的原因是：如果只做应用层 TCP 代理，无法自然支持 ping，也无法覆盖 UDP 协议。项目需要具备更接近真实内网路由的访问体验，因此初版应按三层虚拟网络系统设计。

## 网段映射设计

客户现场真实网段可能冲突。例如客户 A 和客户 B 都使用：

```text
192.168.1.0/24
```

如果直接把真实网段暴露给实施端，会导致路由冲突。因此服务端需要维护“真实网段到平台映射网段”的对照关系。

示例：

| 客户 | 真实网段 | 平台映射网段 |
|---|---|---|
| 客户 A | `192.168.1.0/24` | `100.64.1.0/24` |
| 客户 B | `192.168.1.0/24` | `100.64.2.0/24` |
| 客户 C | `10.10.1.0/24` | `100.64.3.0/24` |

实施人员访问客户 A 的设备：

```text
客户 A 真实设备 IP：192.168.1.10
平台映射访问 IP：100.64.1.10
```

实施端访问：

```text
100.64.1.10
```

系统内部再把目标地址转换回：

```text
192.168.1.10
```

这种设计可以避免多个客户真实网段重复时互相冲突。

## 地址规划

客户映射网段池建议使用：

```text
100.64.0.0/10
```

该地址段常用于运营商级 NAT，普通企业内网直接使用它的概率相对低，适合作为平台虚拟映射地址池。

初版可以按 `/24` 为单位分配映射网段：

```text
192.168.1.0/24 -> 100.64.1.0/24
192.168.2.0/24 -> 100.64.2.0/24
10.10.5.0/24   -> 100.64.3.0/24
```

实施端专用网段建议使用：

```text
100.127.0.0/24
```

示例分配：

```text
服务端虚拟地址：100.127.0.1
实施端 1：100.127.0.2
实施端 2：100.127.0.3
实施端 3：100.127.0.4
```

初期 255 个以内实施端足够使用。

## 访问流程

实施端访问客户现场设备时，数据流转如下：

```text
实施端访问 100.64.1.10
        ↓
实施端 TUN 虚拟网卡接管流量
        ↓
流量进入隧道并发送到服务端
        ↓
服务端查询映射表：100.64.1.0/24 属于客户端 A
        ↓
服务端转发给客户端 A
        ↓
客户端 A 将目标地址还原为 192.168.1.10
        ↓
客户端 A 转发到客户现场真实设备
        ↓
设备响应按原路返回
```

客户端访问其他网段时，需要先经过服务端权限判断。普通客户端默认只能访问自己上报的网段；实施端默认可以访问所有已登记网段。

## 权限模型

初版权限规则如下：

| 角色 | 默认权限 |
|---|---|
| 服务端 | 管理全部节点、网段、映射和权限 |
| 普通客户端 | 默认禁止访问其他客户端网段 |
| 实施端 | 默认允许访问所有已登记网段 |

后台需要支持对普通客户端授权：

```text
1. 禁止访问其他网段。
2. 允许访问指定网段。
3. 允许访问全部网段。
```

实施端初版默认访问全部网段，但后台仍应预留限制访问范围的能力。

## 客户端能力

客户端连接服务端后，需要完成以下工作：

```text
1. 自动识别本机网卡和候选网段。
2. 展示物理网卡、虚拟机网卡、Hyper-V 网卡、VMware 网卡等信息。
3. 用户手动选择要上报的网段。
4. 用户填写网段名称和备注。
5. 将网段信息提交给服务端。
6. 接收服务端分配的映射网段。
7. 参与远端访问流量转发。
```

客户端不应自动开放所有网段。因为客户现场机器可能同时存在办公网、设备网、虚拟机网、Docker 网、VPN 网和无线网，必须由用户确认哪些网段需要暴露。

初版建议只支持 `/24` 网段。这样映射和路由处理更简单，后续再扩展 `/16`、`/23`、`/25` 等复杂网段。

## 实施端能力

实施端连接服务端后，服务端自动分配一个实施端虚拟 IP，例如：

```text
100.127.0.23
```

服务端向实施端下发可访问的映射网段：

```text
100.64.1.0/24
100.64.2.0/24
100.64.3.0/24
```

实施端本机添加对应路由：

```text
访问 100.64.1.0/24 -> 走软件虚拟网卡
访问 100.64.2.0/24 -> 走软件虚拟网卡
访问 100.64.3.0/24 -> 走软件虚拟网卡
```

实施人员通过映射 IP 访问远端设备。后台和客户端界面需要展示映射关系，例如：

```text
客户 A / 生产网 / 192.168.1.0/24 -> 100.64.1.0/24
```

## 平台范围

平台支持分阶段实现。

### 首发版本

```text
服务端：Linux 原生部署、Docker 部署
客户端：Windows、Linux
实施端：Windows、Linux
```

首发版本优先把核心网络能力跑通，包括 TUN、ICMP、TCP、UDP、网段上报、映射分配、权限控制和后台管理。

### 后续版本

```text
服务端：Windows
客户端：Android
实施端：Android
```

Android 端通过系统 VPNService 实现虚拟网络能力。它适合作为实施端临时访问远端设备，也可以作为客户端，但长期稳定暴露客户现场网段时，Windows 或 Linux 更可靠。

## Docker 支持

服务端建议必须支持 Docker。Docker 能降低部署和升级成本，也方便后续私有化部署。

如果服务端涉及 TUN、路由或网络转发，Docker 运行时可能需要额外权限：

```text
--cap-add NET_ADMIN
--device /dev/net/tun
--network host
```

具体参数需要根据最终底层方案确定。

## 开源项目调研

当前调研了几个成熟开源项目，可作为底层网络能力或产品形态参考。

### EasyTier

EasyTier 是一个基于 Rust 的去中心化虚拟专用网络项目，官方说明支持 Windows、macOS、Linux、FreeBSD、Android 和多种 CPU 架构。它提供虚拟组网、NAT 穿透、子网代理、Web 管理、Docker 部署、TCP/UDP/WSS/WireGuard 等能力。[1]

它和本项目需求重合度较高，尤其是子网代理能力。一个节点可以把自身可访问的网段共享给其他节点访问，这与“客户端上报现场网段，实施端访问远端设备”的需求接近。[1]

差异在于 EasyTier 默认偏去中心化，没有天然区分服务端、普通客户端和实施端。本项目需要在其基础上增加中心管理、角色区分、网段映射、权限配置和产品界面。

### NetBird

NetBird 是一个基于 WireGuard 的开源组网平台，具备管理后台、访问控制、外部网络路由、私有 DNS、API、Docker 自托管、Windows/Linux/Android/iOS 等客户端支持。[2]

它的产品化程度较高，适合作为后台管理、访问控制和企业化部署的参考。但 NetBird 体系较重，如果要改成当前项目的网段映射、实施端默认全权限、普通客户端默认隔离等模型，需要深入理解其管理服务、信令服务、Relay 服务和客户端代理架构。

另外，NetBird 仓库中存在 AGPL-3.0 相关授权说明。若后续计划闭源商业化，需要在立项后单独做许可证审查。[2]

### Nebula

Nebula 是 Slack 开源的 Overlay 网络工具，官方说明支持 Linux、Windows、macOS、FreeBSD、iOS 和 Android，可用于连接少量设备，也可以扩展到数万台设备。[3]

它的优势是底层网络模型成熟，支持证书、组、防火墙规则、Lighthouse 发现节点和跨网络互联。Nebula 更适合作为底层网络引擎参考，而不是直接作为完整产品底座，因为它缺少本项目需要的后台、网段登记、映射地址池、客户端 GUI 和实施端管理体验。

### ZeroTier

ZeroTier 是成熟的软件定义网络项目，官方描述其可让设备、虚拟机、容器和应用像处在同一个数据中心或云区域一样互通。它支持端到端加密，大部分流量点对点，也提供 Android 和 iOS 应用。[4]

ZeroTier 的成熟度较高，但其仓库中区分 MPL-2.0 代码和 nonfree 部分，控制器相关内容位于 nonfree/controller。若基于它深度二开，需要重点评估授权、控制器改造和商业化限制。[4]

### Headscale

Headscale 是 Tailscale 控制服务器的开源自托管实现。项目目标是提供一个开源的 Tailscale 控制面，而不是完整替代 Tailscale 客户端。[5]

它适合参考控制面设计，例如节点、路由、用户边界和共享机器模型。但本项目目标是开发自己的统一软件本体，因此 Headscale 不适合作为主底座。

## 初步选型判断

当前建议优先验证 EasyTier，NetBird 作为成熟产品参考，Nebula 作为底层网络实现参考。

| 项目 | 适配度 | 主要价值 | 主要问题 |
|---|---|---|---|
| EasyTier | 高 | 跨平台、子网代理、Android、Docker、Rust、Web/GUI 代码基础 | 默认去中心化，需要补中心管理和角色模型 |
| NetBird | 中高 | 产品完整，后台、权限、路由、客户端齐全 | 体系较重，许可证和二开成本需要评估 |
| Nebula | 中 | 网络层成熟，许可证友好 | 上层产品能力需要自建 |
| ZeroTier | 中低 | 成熟度高，生态完整 | 控制器和授权模式需要谨慎 |
| Headscale | 中低 | 控制面设计参考 | 不适合作为自有客户端产品底座 |

## 初版边界

初版建议明确控制范围：

```text
1. 一个软件本体，支持服务端、客户端、实施端三种模式。
2. 服务端首发支持 Linux 和 Docker。
3. 客户端和实施端首发支持 Windows 和 Linux。
4. Android 放到第二阶段。
5. 底层采用 TUN 虚拟网卡或成熟开源项目的数据面能力。
6. 支持 ICMP、TCP、UDP。
7. 暂时不支持广播和组播。
8. 初版只支持 /24 网段映射。
9. 远端访问统一使用映射 IP。
10. 普通客户端默认禁止访问其他网段。
11. 实施端默认允许访问全部已登记网段。
12. 后台支持节点、网段、映射、权限管理。
```

## PoC 验证目标

在正式开发前，需要先验证开源底座是否满足关键网络能力。建议用 EasyTier 做第一轮验证。

PoC 场景：

```text
服务端：Linux + Docker
客户 A：Windows，暴露 192.168.1.0/24
客户 B：Linux，暴露 192.168.1.0/24
实施端：Windows，访问两个客户的映射网段
```

需要验证：

```text
1. ping 是否可通。
2. TCP 服务是否可访问。
3. UDP 是否可转发。
4. NTP 这类 UDP 协议是否可用。
5. 子网代理是否稳定。
6. 两个客户真实网段相同是否能通过映射层规避冲突。
7. Windows 和 Linux 上路由添加、删除是否可靠。
8. 断线重连后路由和连接是否恢复。
```

## 主要风险

### 网段冲突

多个客户可能使用相同真实网段，这是本项目必须解决的问题。初版采用映射网段隔离，不直接暴露真实网段。

### UDP 会话维护

UDP 没有连接状态，系统需要维护临时会话映射。响应包返回时，需要知道应转回哪个实施端或客户端。初版可采用超时表机制，例如 30 秒或 60 秒无数据后释放。

### Windows 权限

Windows 上创建虚拟网卡、添加路由、修改防火墙规则通常需要管理员权限。安装包、服务启动和权限提示要提前设计。

### Android 后台保活

Android 通过 VPNService 实现虚拟网络能力，但会受到系统省电策略、后台限制和厂商 ROM 影响。Android 不建议作为首发核心平台。

### Docker 网络权限

服务端 Docker 如果需要操作 TUN、路由或转发，可能需要 host 网络和 NET_ADMIN 权限。部署文档必须清晰说明。

### 开源许可证

不同项目许可证差异较大。若项目未来要商业化闭源发布，必须在选型阶段审查许可证，尤其是 AGPL、LGPL、MPL 和 nonfree 目录的约束。

## 立案结论

本项目具备立项价值，且底层网络能力不必完全从零开发。当前最稳妥的方向是：优先基于 EasyTier 做技术验证，借鉴 NetBird 的后台管理和权限模型，参考 Nebula 的底层组网实现，再结合本项目自有的网段映射、角色权限和实施端访问模型形成产品。

项目的核心差异点不在“能不能组网”，而在于“如何面向客户现场和实施场景，把网段登记、映射访问、默认隔离和实施端全局访问做成简单可靠的产品体验”。
